# DataPivot 数据智能分析系统 — 技术方案说明

## 一、项目概述

DataPivot 是一个面向车辆报警数据的 **AI 驱动型数据分析系统**。用户通过自然语言对话即可完成数据查询、分析与可视化，无需掌握 SQL 或数据透视等专业技能。系统同时提供可视化下钻与拖拽配置作为辅助手段。

### 核心能力

| 能力 | 说明 |
|------|------|
| 🤖 **AI 对话分析** | **核心能力**。用户用自然语言提问，AI 自动理解语义、探索数据库结构、生成 SQL、执行查询、推荐图表、输出洞察。多轮对话支持追问与上下文记忆 |
| 🔍 链路追踪 | AI 分析过程全透明可视化：LLM 调用链、工具调用、SQL 执行、耗时统计，便于调试与可信度评估 |
| 📋 数据下钻 | 人员 → 任务 → 规则 → 报警 → 信号，5 层分层下钻，波形图展示 |
| 📊 辅助配置 | 字段拖拽配置 + 图表/表格双视图（作为 AI 对话的补充手段） |

## 三、AI 对话分析（核心功能）

### 3.1 用户交互流程

```
用户在聊天框输入自然语言问题
  例: "按车型统计报警数量"
       │
       ▼
SSE 流式实时推送思考过程到前端  ← 用户可实时看到 AI 在做什么
  ├── 🟢 status  → "正在分析查询意图..."
  ├── 💭 thinking → 中间推理步骤
  ├── 🔧 tool    → "正在执行 SQL 查询..."
  ├── 📊 result  → 最终数据 + 图表 + 追问建议
  └── ✅ done    → 完成
       │
       ▼
AI 自动完成:
  1. 理解自然语言 → 2. 探索数据库 Schema →
  3. 编写 SQL → 4. 执行查询 → 5. 推断图表类型 →
  6. 生成 Vega-Lite 图表 → 7. 输出文字洞察 →
  8. 推荐 3 个追问
       │
       ▼
用户可继续追问，AI 自动继承对话上下文
  例: "只看货车的" → AI 理解是在上一个结果上过滤
```

### 3.2 多轮对话机制

```
每次对话通过 session_id 关联:

  用户 A 的第 1 问 → session_id="abc" → Thread_1 → 结果
  用户 A 的第 2 问 → session_id="abc" → Thread_1 → 继承上下文
  用户 B 的第 1 问 → session_id="xyz" → Thread_2 → 独立的会话

  无 session_id 时自动分配新的 session_id
  所有线程保存在内存 Dict 中，进程重启后丢失
```

### 3.3 SSE 流式技术方案

```
前端 fetch → POST /api/chat/stream

后端事件流:
  event: status   data: {"phase":"start","message":"🔍 正在分析..."}
  event: thinking data: "正在探索数据库结构..."
  event: tool     data: "[tool_call: 执行SQL]"
  event: status   data: {"phase":"thinking","message":"📊 正在生成图表..."}
  event: result   data: {data, chart_spec, answer, followups}
  event: done     data: {"message":"✅ 完成"}

前端解析方式:
  原生 ReadableStream → 按行解析 SSE 协议 →
  按 event type 分发给不同 UI 组件
```

### 3.4 边界条件

| 场景 | 处理方式 |
|------|---------|
| API Key 未配置 | 直接报错提示配置方式，不影响其他功能 |
| API Key 不合法 | 捕获异常，日志记录完整调用栈，SSE 推送错误 |
| 数据库查询结果为空 | AI 返回"未能查询到相关数据，请尝试换个问法" |
| timedelta 类型列 | 自动转为秒数 (float64)，否则无法渲染图表 |
| numpy 数据类型 | 递归转换为 Python 原生类型，确保 JSON 序列化 |
| DataFrame 嵌套数组 | 自动 explode 展开为多行 |
| SSE 连接中断 | 前端 AbortController 取消，后端继续执行但结果丢弃 |
| 对话线程无过期机制 | 当前无限增长，建议后续加入 TTL 淘汰策略 |
| 流式输出行 > 600字符 | 跳过不发送，避免前端卡顿 |
| 无 API Key 时不初始化 AI 引擎 | 节省资源，避免无意义的失败 |

## 四、AI 提示词（Prompt）体系

AI 对话引擎的运行依赖于多阶段提示词模板，覆盖从 SQL 生成到图表推荐的完整流程。以下列出每个环节的核心提示词内容（英文原文 + 中文翻译）。

### 4.1 AI 系统提示词（System Prompt）

**位置：** `agent/executors/lighthouse/system_prompt.jinja` — Jinja2 模板

AI 在每轮对话开始时注入此 System Prompt，其中包含数据库 Schema、上下文信息和当前日期。

**英文原文（Jinja2 模板）：**

```
You are a "Databao" agent that has direct access to the database.
You generate SQL requests, which are executed on a DB client with no changes.
User can connect several databases and DataFrames to your internal DuckDB instance.
DataFrames are available as tables with "temp.main" prefix.
The task is to request all necessary data and answer the user question.
You can answer with:
- text (using plain text)
- a table (using SQL requests and query_id parameter of submit_result tool)
- a plot (using visualization parameter of submit_result tool)
or a combination of these.

Today's date is: {{ date }} (YYYY-MM-DD).

# Instructions:
- Solve complex requests step by step
  - Briefly describe each step before running the query and explain why you are doing it.
  - If several similar tables or columns can be used, try both options, determine root cause
    of the difference in results and choose the best one.
- Get DB schema in the 'Database schema' section. Don't waste tool call for it.
- Pay attention to SQL dialect specific commands (DuckDB is used)
- Cross joins are allowed only for tables that are guaranteed small (< 5 rows).
- When calculating percentages like (a - b) / a * 100, you must make multiplication first
  to prevent number rounding. Use 100 * (a - b) / a.
- When comparing an unfinished period like the current year to a finished one like last year,
  use the same date range. Never compare unfinished periods to finished one.
- Make sure the submitted result answers the user's question and it is not empty
  - Result description should contain definitions being used, important decisions
    and analysis of resulting data
  - Leave visualization prompt empty if you don't want to visualize the result.
    Table with few values or table with heterogeneous data don't need visualization
  - Time series require visualization
- The user will see only the submitted result - final SQL and DataFrame.
  The user will not see intermediate results
- Use less than {{ tool_limit }} tool calls before submitting the result

# Database schema
## Full Database schema
{{ db_schema }}
```

**中文翻译：**

```
你是一个可以直接访问数据库的 AI 分析助手。
你生成 SQL 请求，这些请求会在数据库客户端上直接执行。
用户可以连接多个数据库和 DataFrame 到内部 DuckDB 实例。
DataFrame 以 "temp.main" 前缀的表形式提供。
你的任务是获取所有必要数据并回答用户问题。
你可以通过以下方式回答：
- 文本（纯文本）
- 表格（使用 submit_result 工具的 query_id 参数）
- 图表（使用 submit_result 工具的 visualization 参数）
或组合使用。

今天的日期是：{{ date }}（YYYY-MM-DD）。

# 使用说明：
- 复杂请求要逐步解决
  - 在执行查询前简要描述每个步骤并解释原因。
  - 如果有多个相似的表或列，尝试所有选项，找出结果差异的原因并选择最佳方案。
- 数据库架构在"Database schema"部分，不要为此浪费工具调用。
- 注意 SQL 方言特定命令（使用的是 DuckDB）。
- 交叉连接仅允许用于保证小数据量（< 5 行）的表。
- 计算百分比如 (a - b) / a * 100 时，必须先做乘法防止数值舍入。使用 100 * (a - b) / a。
- 比较多整年的同比周期时，使用相同的日期范围。不要比较未完成周期与已完成周期。
- 确保提交的结果回答了用户的问题且不为空。
  - 结果描述应包含所使用的定义、重要决策和对结果数据的分析。
  - 如果不需要可视化，留空 visualization prompt。少量数值或异构数据不需要可视化。
  - 时间序列数据需要可视化。
- 用户只会看到提交的最终结果——SQL 和 DataFrame。用户不会看到中间结果。
- 在提交结果前，使用不得超过 {{ tool_limit }} 次工具调用。

# 数据库架构
## 完整数据库架构
{{ db_schema }}
```

**运行时注入变量：**

| 变量 | 来源 | 说明 |
|------|------|------|
| `{{ date }}` | `get_today_date_str()` | 当前日期，格式 "Monday, 2026-06-29" |
| `{{ db_schema }}` | `inspect_database_schema()` | 数据库完整 Schema：表名(列名: 类型, ...) |
| `{{ tool_limit }}` | `recursion_limit // 2` | 最大工具调用次数上限 |
| `{{ db_types }}` | 数据源配置 | 已连接的数据库类型列表 |
| `{{ context }}` | `build_context_text()` | 用户提供的额外上下文说明 |
| `{{ dce_search_enabled }}` | 配置开关 | 是否启用搜索上下文工具 |

### 4.3 AI 工具设定（Tools Configuration）

AI 智能体在执行任务时可调用以下工具，工具的描述和参数作为提示词的一部分注入到 LLM 上下文中。

#### 工具 1：`run_sql_query` — SQL 查询执行

| 属性 | 说明 |
|------|------|
| **用途** | 在 DuckDB 数据库中执行 SELECT 查询 |
| **注册方式** | `@tool(description=RUN_SQL_QUERY_TOOL_DESCRIPTION)` |
| **调用限制** | 整个对话中不得超过 `{{ tool_limit }}` 次（通常为 `recursion_limit / 2`） |
| **返回给 LLM 的内容** | CSV 格式的前 12 行数据 + 自动生成的 `query_id` |

**工具描述（英文原文，即 LLM 看到的）：**

```
Run a SELECT SQL query in the database. Returns the first 12 rows in csv format.

Args:
    sql: SQL query
```

**中文翻译：**

```
在数据库中执行 SELECT SQL 查询。返回前 12 行的 CSV 格式数据。

参数：
    sql: SQL 查询语句
```

**实际返回数据结构（`tool_executor_node` 处理后返回给 LLM 的内容）：**

```
成功时返回给 LLM 的文本内容:
query_id='{自动生成的ID}'
{CSV格式数据}

失败时返回给 LLM 的文本内容:
{异常信息}
Tool: run_sql_query, Args: {参数}
```

**底层 `_run_sql_query()` 函数返回的原始字典（被 `tool_executor_node` 加工后存储）：**

```python
# 成功时:
{
    "df": pd.DataFrame,       # 完整查询结果 DataFrame
    "sql": "SELECT ...",       # 实际执行的 SQL
    "csv": "...",              # CSV 格式的前 12 行（用于返回给 LLM）
    "markdown": "..."          # Markdown 格式的前 12 行
}

# 失败时:
{
    "error": "异常信息"        # 异常堆栈字符串
}
```

**执行流程：**

```
LLM 输出 tool_call("run_sql_query", {"sql": "SELECT ..."})
  → tool_executor_node 收到调用
  → tool.invoke(args) 调用 _run_sql_query()
  → _run_sql_query() 返回 {"df": DataFrame, "sql": "...", "csv": "..."}
  → tool_executor_node 生成 query_id = "{消息索引}-{工具调用索引}"
  → 将 csv 内容 + query_id 返回给 LLM
  → 将完整结果存储到 state.query_ids[query_id] 供后续 submit_result 使用
```

---

#### 工具 2：`submit_result` — 提交最终结果

| 属性 | 说明 |
|------|------|
| **用途** | 将查询结果提交给用户，必须是最后一个工具调用 |
| **注册方式** | `@tool(parse_docstring=True)`——docstring 作为工具描述 |
| **调用后** | LLM 循环终止，控制权返回给用户 |

**工具描述（英文原文，即 LLM 看到的）：**

```
Call this tool with the ID of the query you want to submit to the user.
This will return control to the user and must always be the last tool call.
The user will see the full query result, not just the first 12 rows. Returns a confirmation message.

Args:
    query_id: The ID of the query to submit (query_ids are automatically generated when you run queries).
    result_description: A comment to a final result. This will be included in the final result.
    visualization_prompt: Optional visualization prompt. If not empty, a Vega-Lite visualization agent
        will be asked to plot the submitted query data according to instructions in the prompt.
        The instructions should be short and simple.
```

**中文翻译：**

```
调用此工具来提交你想要呈现给用户的查询结果。
此调用将控制权返还给用户，且必须是最后一个工具调用。
用户将看到完整的查询结果，而不仅仅是前 12 行。返回确认消息。

参数：
    query_id: 要提交的查询 ID（执行查询时自动生成）。
    result_description: 最终结果注释，会包含在最终结果中。
    visualization_prompt: 可选的可视化描述。如果不为空，Vega-Lite 可视化引擎将根据
        描述中的指令对提交的查询数据绘图。指令应简短明了。
```

**submit_result 的执行校验规则：**

```
1. 如果 submit_result 与其他工具同时调用 → 驳回，要求唯一调用
2. 如果 query_ids 为空（没有任何查询被执行过）→ 驳回
3. 如果 query_id 不存在于 query_ids 中 → 驳回，告知可用 ID 列表
4. 如果 query_id 对应的查询结果中没有 DataFrame → 驳回
5. 校验通过 → 从 state.query_ids[query_id] 中提取 sql 和 df
               → 设置 state 中的 sql、df、visualization_prompt
               → ready_for_user = true → LangGraph 停止循环
```

#### 工具 3：`search_context` — 搜索上下文信息（条件启用）

| 属性 | 说明 |
|------|------|
| **用途** | 在进行最终 SQL 查询前，搜索表/列的元数据描述信息 |
| **启用条件** | 仅当数据源（Domain）支持上下文搜索时可用，否则不加入工具列表 |

```
Remember to use the search_context tool to find relevant context
(e.g., table and column descriptions) before writing your final SQL query.
```

（中文：记得在编写最终 SQL 前使用 search_context 工具查找相关上下文，如表和列的描述信息。）

#### LLM 模型配置

AI 智能体使用的 LLM 参数由 `LLMConfig` 控制，在 `chat.py` 中初始化：

| 参数 | 说明 | 当前值 |
|------|------|--------|
| `name` | 模型名称 | `deepseek-chat`（有 DeepSeek Key 时）或 `gpt-4o-mini`（回退） |
| `temperature` | 生成随机性 | `0.1` |
| `api_base_url` | API 端点 | 有 DeepSeek Key 时使用 `DEEPSEEK_BASE_URL`，否则使用 OpenAI 默认 |
| `use_responses_api` | 是否使用 Responses API | DeepSeek 时关闭（仅支持 /chat/completions） |
| `recursion_limit` | LangGraph 递归上限 | `200`（在 `chat.py` 中配置） |
| `parallel_tool_calls` | 是否允许并行工具调用 | OpenAI 模型时启用，DeepSeek 时关闭 |

**LLM 调用链路：**

```
LLMConfig.new_chat_model()
  → 创建 ChatOpenAI / ChatDeepSeek 实例
  → model_bind_tools() 绑定工具到模型
  → llm_node: chat(messages, model_config, model_with_tools)
  → 返回 LLM 生成的文本或 tool_call
```

### 4.5 图表生成提示词（Vega-Lite Spec Generation）

**位置：** `edaplot/vega_chat/prompts.py` — `get_system_prompt_v2()`

**英文原文（核心部分）：**

```
Your task is to generate a valid Vega-Lite chart specification for the given dataset and user utterance delimited by <>.
The user may want to refine the chart with a follow-up request. The refined chart must include ALL previously given information by the user.

Do the following step by step and always include all steps in your response. All explanations must be in {language_spec}.
1. Explain the visualization goal for the chart you are about to generate.
   Explain whether filters should be applied, which chart type and columns should be used, and what transformations are necessary.
2. Determine whether the request is relevant to the given dataset. Provide a brief one sentence rationale.
```

### 4.6 图表类型推断逻辑（Rule-based Fallback）

当图表生成引擎不可用时，系统使用规则推断图表类型：

```
位置: chat.py — _infer_chart_type()

逻辑:
  1. 关键词匹配:
     "趋势"/"时间"/"变化"/"走势" → 折线图
     "占比"/"比例"/"百分比"/"分布" → 饼图 (数据≤10行) 或 柱状图
     "散点"/"相关" → 散点图
  2. 数据特征:
     有时间列 + 数值列 → 折线图
     单分类 + 单数值 + 数据≤6行 → 饼图
     其他 → 柱状图
```

## 八、数据库设计

### 8.1 表结构

```sql
TM_RMU_PS_TASK (714行)           — 任务基本信息
TM_RMU_PS_TASK_VEHICLETYPE (714行) — 车型 (1:1)
TM_RMU_PS_TASK_VEHICLE (13,163行)  — 车辆 (1:N)
TM_RMU_PS_TASK_RULE (3,109行)      — 规则 (1:N)，含规则类型
TL_RMU_PS_TASK_RULE_RESULT (301行)  — 报警结果 (1:1)
TL_RMU_PS_TASK_RULE_RESULT_SIGNAL (23,127行) — 信号值 (1:N，243种信号)
```

### 8.2 查询引擎设计

数据库为只读模式，所有查询通过 CTE（WITH 子句）实时 JOIN 6 张表为扁平化宽表：

```
WITH v AS (6 表 JOIN) SELECT * FROM v WHERE ... GROUP BY ...
```

信号值（243 种动态列）通过标量子查询实时提取，不预关联：

```sql
SUM((SELECT TRY_CAST(s.VALUE AS DOUBLE) FROM 信号表
     WHERE s.报警ID = v.报警ID AND s.信号名称 = 'XXX'))
```

**智能聚合降级：** SUM/AVG 作用于文本或时间列时自动降级为 COUNT，不中断查询。
