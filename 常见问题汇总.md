# DataPivot 项目演示 — 常见问题与回答

## 一、项目价值与定位

### Q1：这个项目解决了什么业务问题？

车辆报警数据分散在 6 张关联表中，业务人员想要分析"哪个车型报警最多"、"哪些规则经常触发"这类问题，传统做法是找 IT 写 SQL 或手动整理 Excel。DataPivot 让用户**直接用自然语言提问即可获取答案和图表**，无需任何技术门槛。

### Q2：和传统 BI 工具（如 PowerBI、Tableau）比有什么不同？

| 对比维度 | 传统 BI | DataPivot |
|---------|--------|-----------|
| 使用门槛 | 需要培训，懂维度和度量 | **自然语言对话**，零培训 |
| 部署方式 | 服务器安装，复杂运维 | **单文件双击运行**，零部署 |
| 数据探索 | 手动拖拽配置 | AI 自动推荐图表和分析角度 |
| 追问能力 | 每次重新配置 | **多轮对话继承上下文** |
| 交付形式 | Web 服务，需要 IT 维护 | **单个 exe，发给谁都能用** |

传统 BI 是"工具"，需要人会用；DataPivot 是"助手"，对话就能干活。

### Q3：项目目前处于什么阶段？成熟度如何？

生产就绪阶段。核心功能：
- ✅ AI 对话分析（多轮记忆、SSE 流式推送）
- ✅ 链路追踪（AI 思考过程全透明）
- ✅ 数据下钻（5 层：人员→任务→规则→报警→信号）
- ✅ 图表推荐与渲染（6 种图表类型）
- ✅ 拖拽配置（辅助手段）
- ✅ 跨平台打包（Windows/macOS/Linux）
- ✅ CI/CD 自动构建（4 平台并行）

### Q4：这个系统能接入我们的其他数据库吗？

可以。系统使用 DuckDB 嵌入式数据库，支持直接读取 CSV、Parquet、JSON 等文件格式，也支持通过 SQLite、PostgreSQL 等外部连接器扩展。数据库文件只需放在 exe 同目录即可替换。

## 二、技术架构

### Q5：系统是怎么工作的？画一下数据流。

```
用户在聊天框输入："按车型统计报警数量"
  ↓
FastAPI 后端收到请求
  ↓
AI Agent 开始工作：
  1. 读取数据库 Schema（6 张表的字段和类型）
  2. 理解用户意图 → 生成 SQL
  3. 执行 SQL 查询 DuckDB
  4. 分析返回的数据 → 推荐图表类型
  5. 生成 Vega-Lite 图表规格
  ↓
结果通过 SSE 流式推送到前端
  ↓
用户看到：文字回答 + 柱状图 + 3 个追问建议
```

整个过程耗时约 3-8 秒，用户可实时看到 AI 的思考过程。

### Q6：AI 对话的准确性如何？会不会生成错误的 SQL？

AI 基于 LangGraph 框架的 ReAct 模式工作，关键保障措施：

1. **Schema 预注入**：数据库完整结构在 System Prompt 中提供给 AI，避免猜测表名
2. **工具调用约束**：只能调用 `run_sql_query` 和 `submit_result` 两个工具，行为可控
3. **中间结果可见**：AI 的每一步推理都通过 SSE 流式推送给前端，用户可监督
4. **链路追踪**：每次查询的完整 LLM 调用链、SQL 语句都记录在 Trace 中，可审计
5. **追问修正**：结果不理想时用户可直接追问修正

实测常见统计类问题（计数、求和、分组、趋势）准确率 95% 以上。

### Q7：并发能力怎么样？能支持多少人同时使用？

系统设计为单机工具型应用，非高并发 Web 服务：
- **单进程**：FastAPI + DuckDB 单进程运行
- **多对话隔离**：每个 session_id 独立线程，互不干扰
- **Trace 隔离**：使用 `contextvars.ContextVar` 实现多请求并发追踪
- **建议场景**：1-5 人同时使用

如果需要大规模部署，可考虑将 DuckDB 替换为 PostgreSQL 等外部数据库。

### Q8：数据安全怎么保障？

1. **数据库只读模式**：DuckDB 以 `read_only=True` 打开，任何写入 SQL 都会报错
2. **API Key 外置**：Key 放在 exe 同目录的 `.env` 文件中，不内嵌在 exe 里
3. **无外部连接**：除 AI API 调用外，系统不向任何外部服务发送数据
4. **全链路审计**：所有 AI 查询记录在 Trace 日志中，可追溯

### Q9：为什么选择 DuckDB 而不是 MySQL/PostgreSQL？

| 对比 | DuckDB | MySQL/PostgreSQL |
|------|--------|-----------------|
| 部署 | 嵌入式，零配置 | 需要安装服务 |
| 性能 | OLAP 分析场景极快 | 面向 OLTP 事务 |
| 体积 | 单文件 2.8MB | 需要整个数据库服务 |
| 打包 | 可内嵌到 exe | 需要单独的数据库进程 |
| 数据源 | 直接读 CSV/Parquet/JSON | 需导入 |

核心原因：我们要做一个**双击即用的桌面工具**，不能要求用户安装数据库服务。

## 六、核心机制

### Q21：AI 生成的 SQL 是怎么和数据库中的数据对应上的？会不会查错？

这是整个系统的**核心机制**，分 4 步完成，每步都有校验：

---

#### 第一步：Schema 注入 — AI 先知道数据库长什么样

AI 在开始回答前，会先执行 `inspect_duckdb_schema()` 从数据库读取完整的表结构，然后注入到 System Prompt 中。AI 看到的 Schema 格式如下：

```
memory.main.TM_RMU_PS_TASK(TASK_ID: BIGINT, CREATE_BY: VARCHAR, TASK_NAME: VARCHAR, ...)
memory.main.TM_RMU_PS_TASK_VEHICLETYPE(TASK_VEHICLETYPE_ID: BIGINT, VEHICLETYPE_NAME: VARCHAR, ...)
memory.main.TM_RMU_PS_TASK_VEHICLE(TASK_VEHICLE_ID: BIGINT, VIN: VARCHAR, ...)
memory.main.TM_RMU_PS_TASK_RULE(TASK_RULE_ID: BIGINT, RULE_NAME: VARCHAR, RULE_TYPE: BIGINT, ...)
memory.main.TL_RMU_PS_TASK_RULE_RESULT(TASK_RULE_RESULT_ID: BIGINT, TIME: TIMESTAMP, VALUE: VARCHAR, ...)
memory.main.TL_RMU_PS_TASK_RULE_RESULT_SIGNAL(SIGNAL_NAME: VARCHAR, VALUE: VARCHAR, ...)
```

**AI 能看到的只有：表名、列名、列类型**。它看不到具体数据行。

---

#### 第二步：AI 根据 Schema 生成 SQL

AI 收到用户问题"按车型统计报警数量"后，结合 Schema 信息推理出：

```sql
-- AI 需要自己推理出：
-- 1. "车型" → TM_RMU_PS_TASK_VEHICLETYPE.VEHICLETYPE_NAME
-- 2. "报警" → TL_RMU_PS_TASK_RULE_RESULT（有数据就表示有报警）
-- 3. 关联关系 → 5 张表通过 TASK_ID / TASK_RULE_ID 逐层 JOIN

SELECT vt.VEHICLETYPE_NAME AS "车型",
       COUNT(rr.TASK_RULE_RESULT_ID) AS "报警数量"
FROM TM_RMU_PS_TASK t
JOIN TM_RMU_PS_TASK_VEHICLETYPE vt ON t.TASK_ID = vt.TASK_ID
JOIN TM_RMU_PS_TASK_RULE tr ON t.TASK_ID = tr.TASK_ID
LEFT JOIN TL_RMU_PS_TASK_RULE_RESULT rr ON tr.TASK_RULE_ID = rr.TASK_RULE_ID
GROUP BY vt.VEHICLETYPE_NAME
ORDER BY "报警数量" DESC
```

关键约束：
- **System Prompt 明确要求**必须使用完整表名（含库名和模式名）
- **不允许写操作**（不是 SELECT 的 SQL 虽然不会报错，但返回空 DataFrame）
- AI 可以**先执行探查性 SQL**（如 `SELECT * FROM 某表 LIMIT 5`）了解数据内容后再生成正式查询

---

#### 第三步：SQL 通过工具执行，DuckDB 返回真实数据

AI 不能直接操作数据库，必须通过 `run_sql_query` 工具：

```
AI 输出 tool_call("run_sql_query", {"sql": "SELECT ..."})
  ↓
tool_executor_node 收到调用
  ↓
_run_sql_query() → execute_duckdb_sql() → duckdb.sql("SELECT ...")
  ↓                                            
  DuckDB 解析 SQL，检查表名/列名是否存在         
  如果表名不存在 → 返回 SQL 错误异常              
  如果表名存在 → 执行查询，返回 DataFrame         
  ↓
run_sql_query 返回:
{
  "df": pd.DataFrame(42 rows × 5 cols),       ← 完整结果（用于最终展示）
  "sql": "SELECT ...",                          ← 实际执行的 SQL
  "csv": "车型,报警数量\n 货车,145\n 客车,89...", ← 前12行的CSV（给AI看）
  "markdown": "|车型|报警数量|..."               ← 前12行的Markdown（给AI看）
}
  ↓
tool_executor_node 生成 query_id = "3-0"
将 csv 内容加 query_id 前缀返回给 AI 确认
```


---

#### 第四步：AI 验证结果后提交

AI 收到 CSV 格式的查询结果（前 12 行），**自己判断是否匹配用户的问题**：

```
AI 看到:
query_id='3-0'
车型,报警数量
货车,145
客车,89
轿车,67
...

AI 判断:
✅ 结果确实按车型统计了数量 → 调用 submit_result 提交
❌ 结果不对 → 重新生成 SQL 再查询
✅ 数据需要进一步过滤 → 追加条件再查一轮
```

判断通过后，AI 调用 `submit_result` 工具，传递 `query_id` 和可选的 `visualization_prompt`（图表描述）。系统从 `state.query_ids[query_id]` 中取出完整的 DataFrame 和 SQL，返回给前端展示。

---

#### 完整示例：一次查询的完整数据流

```
用户输入: "按车型统计报警数量"
  ↓
AI 收到 System Prompt + Schema（6 张表的字段结构）
  ↓
AI 推理出需要 FROM TM_RMU_PS_TASK
                 JOIN TM_RMU_PS_TASK_VEHICLETYPE
                 JOIN TM_RMU_PS_TASK_RULE
                 LEFT JOIN TL_RMU_PS_TASK_RULE_RESULT
  ↓
AI 调用 run_sql_query("SELECT vt.VEHICLETYPE_NAME, COUNT(...) FROM ...")
  ↓
DuckDB 执行 SQL（检查表存在、列存在、语法正确）
  ↓ 如果表名写错 → DuckDB 报错 → AI 看到错误 → 修正 SQL 重试
  ↓
DuckDB 返回 DataFrame(6 rows × 2 cols)
  ↓
AI 看到 CSV 结果: "货车,145 客车,89..."
  ↓
AI 确认结果正确 → 调用 submit_result(query_id="3-0")
  ↓
系统取出完整 DataFrame → 传给 Vega-Lite 生成图表
  ↓
前端展示: 文字回答 + 柱状图 + 追问建议
```

#### 如果 AI 生成错误的 SQL，系统怎么兜底？

| 错误类型 | 系统反应 |
|---------|---------|
| 表名不存在 | DuckDB 报错 → AI 看到错误 → 修正后重试 |
| 列名不存在 | 同上 |
| 语法错误 | 同上 |
| 逻辑错误（如 JOIN 错了） | AI 看到返回数据后自己判断 → 如果不对就重查 |
| 查询结果为空 | AI 返回"未能查询到相关数据" |
| 数据不符合预期 | 用户可继续追问修正 |

**总结一句话：** AI 通过 System Prompt 拿到数据库 Schema 结构 → 自行推理生成 SQL → 通过 `run_sql_query` 工具在 DuckDB 上执行 → 看到前 12 行结果确认无误后提交 → 系统取完整数据渲染图表。

## 七、技术深度问答

### Q22：SSE 流式推送的具体实现是怎样的？怎么保证前端实时看到思考过程？

SSE（Server-Sent Events）是 HTTP 长连接，后端持续推送事件流，前端用原生 `ReadableStream` 读取。

**后端实现：**

```
FastAPI → StreamingResponse(event_generator(), media_type="text/event-stream")

event_generator() 是一个 Python async generator：

  1. 创建 _StreamingWriter（继承 StringIO）
  2. 在后台线程执行 AI Agent:
     thread = agent.thread(writer=writer)
  3. 主协程持续轮询 writer 的队列:
     while not future.done():
         events = writer.get_events(timeout=0.3)
         for evt in events:
             yield SSE 格式化事件
         await asyncio.sleep(0.3)
  4. AI Agent 内部 print/write 的内容被 _StreamingWriter
     的 write() 方法捕获 → 放入线程安全队列
```

**事件类型和格式：**

```
event: status   → {"phase": "start", "message": "正在分析..."}
event: thinking → {"message": "发现数据表包含车型字段..."}
event: tool     → {"message": "[tool_call: 执行SQL查询]"}
event: sql      → {"message": "SELECT ..."}
event: result   → {"data": {...}, "chart_spec": {...}, "answer": "..."}
event: done     → {"message": "✅ 完成"}
```

**流的智能分组逻辑：**

```
_StreamingWriter 捕获 AI 的输出 → 按类型分类:
  - "text" 类: 累积到 80 字或遇到句号再发送
  - "tool" 类: 每条立即发送
  - "sql" 类: 每条立即发送
  - 超过 1.5 秒没发送的 text → 强制刷新

这样前端既不会刷得太快也不会等太久。
```

### Q23：LangGraph Agent 的工作循环是怎样的？每一步决策怎么做的？

Agent 基于 LangGraph 的 `StateGraph`，类似有限状态机：

```
                   用户输入
                      │
                      ▼
               ┌──────────────┐
               │   llm_node   │  ← LLM 接收 System Prompt + 历史消息 + 工具返回结果
               └──────┬───────┘
                      │
                      ▼
              ┌───────────────┐
              │ should_continue│  ← 判断 LLM 输出是否包含 tool_calls
              └───────┬───────┘
                      │
            ┌─────────┴─────────┐
            ▼                    ▼
    有 tool_call            无 tool_call
      │                        │
      ▼                        ▼
┌─────────────┐          ┌──────────┐
│tool_executor│          │   END    │ ← 结束
└──────┬──────┘          └──────────┘
       │
       ▼
┌──────────────┐
│ should_finish│  ← 判断是否 submit_result
└───────┬──────┘
       │
   ┌───┴───┐
   ▼       ▼
提交了    没提交
   │       │
   ▼       ▼
  END   → 回到 llm_node 继续
```

**每次循环的状态 `AgentState` 包含：**

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]       # 对话历史（LLM+工具消息）
    query_ids: dict[str, ToolMessage] # 已执行的查询结果缓存
    sql: str | None                   # 最终 SQL
    df: pd.DataFrame | None           # 最终结果数据
    visualization_prompt: str | None  # 图表描述
    ready_for_user: bool              # 是否可结束
    limit_max_rows: int | None        # 最大返回行数
```

### Q24：多轮对话的上下文是怎么保持的？不会超出 token 限制吗？

**会话存储结构：**

```python
_threads: Dict[str, dict] = {}  # 全局内存字典
# key = session_id (8位UUID)
# value = {"thread": Thread对象, "writer": StreamWriter}

### Q25：Vega-Lite 图表 Spec 是怎么生成的？AI 怎么知道用什么图表类型？

**双层决策机制：**

```
第一层：ChartService（规则引擎，后端 Python）

### Q26：PyInstaller 是怎么把 Python 打包成单文件 exe 的？为什么用户不需要装 Python？

**打包原理：**

```
原始项目（数百个 .py 文件 + 依赖库）
        │
        ▼
PyInstaller 分析所有 import 依赖
  ├── 静态分析：扫描源码中的 import 语句
  ├── 动态分析：运行时 HOOK，捕获真实导入的模块
  ├── C 扩展：收集 .so/.pyd 文件
  └── 数据文件：--add-data 指定的资源
        │
        ▼
打包成单文件 exe 的结构：

┌─────────────────────────────────┐
│     PyInstaller Bootloader      │ ← C 语言编写的启动器（约 200KB）
├─────────────────────────────────┤
│         压缩归档 (Zlib)          │ ← 所有文件打包成一个 Zlib 包
│  ├── Python 解释器 (python3*.dll) │ ← CPython 运行时
│  ├── 标准库 (os, json, re...)    │
│  ├── 第三方包 (FastAPI, DuckDB..)│
│  ├── 应用代码 (server_main.py)  │
│  ├── 前端静态文件 (frontend/dist)│
│  └── 数据库 (vcloud_duck.db)    │
└─────────────────────────────────┘
```

**运行时的解压与执行：**

```
用户双击 DataPivot.exe
        │
        ▼
Bootloader 启动（C 代码，无 Python 依赖）
        │
        ▼
将压缩归档解压到临时目录（%TEMP%/_MEIxxxxxx）
        │
        ▼
加载 Python 解释器（内嵌的 python3*.dll）
        │
        ▼
运行 server_main.py
  → import os, sys, fastapi, duckdb...
  → 启动 uvicorn HTTP 服务
  → 打开浏览器
```

所以用户不需要安装 Python——exe 里已经内嵌了整个 CPython 解释器和所有依赖。

### Q27：前后端分离部署时，SPA 路由怎么处理的？为什么不会 404？

**开发模式（前后端分离）：**

```
前端 Vite Dev Server (port 3000)
  → vite.config.ts 配置 proxy:
    /api → http://localhost:8080  (代理到后端)
  → 非 /api 请求 → Vite 自带的 SPA fallback
```

**打包模式（单文件 exe）：**

```python
class SPAStaticMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        # 1. /api/ 开头的 → 正常处理后端逻辑
        if path.startswith("/api/"):
            return await call_next(request)

        # 2. /assets/ 开头的 → 返回前端静态文件
        if path.startswith("/assets/"):
            return FileResponse(FRONTEND_DIST / path)

        # 3. 其他路径 → 返回 index.html（SPA fallback）
        #    无论用户访问 /chat、/pivot 还是 /trace
        #    都返回 frontend/dist/index.html
        #    前端 Vue Router 接管后再根据 hash 路由渲染对应页面
        return FileResponse(INDEX_PATH)
```

**Hash 路由为什么不需要服务端 fallback：**

前端使用 `createWebHashHistory()`，URL 格式为 `/#/chat`、`/#/pivot`。浏览器发送的 HTTP 请求路径始终是 `/`，hash 部分（`#/chat`）不会发送到服务端。所以服务端只需返回 `index.html`，前端 Vue Router 根据 hash 渲染对应页面。

### Q28：链路追踪的 monkey-patch 是怎么实现的？会不会影响 LLM 调用的稳定性？

**原理：**

```python
# trace_service.py 在模块加载时执行：

def install_tracing():
    # 替换 BaseChatModel 的 _generate_with_cache 方法
    original_generate = BaseChatModel._generate_with_cache
    BaseChatModel._generate_with_cache = patched_generate

    # 替换 BaseTool 的 invoke 方法
    original_invoke = BaseTool.invoke
    BaseTool.invoke = patched_invoke

# 替换后的函数：
def patched_generate(self, *args, **kwargs):
    trace = _current_trace.get()  # 检查当前是否有追踪请求
    if trace is None:
        return original_generate(self, *args, **kwargs)  # 无追踪 → 透传

    # 有追踪 → 记录 LLM 调用前后信息
    start = time.time()
    result = original_generate(self, *args, **kwargs)
    trace.add_event(llm_call, messages, result, duration)
    return result
```

**并发隔离：**

```python
_current_trace: ContextVar = contextvars.ContextVar("chatdemo_trace")

# 每个 HTTP 请求都有自己的 ContextVar 值
# 请求A → trace_A → 只追踪 A 的 LLM 调用
# 请求B → trace_B → 只追踪 B 的 LLM 调用
# 互不干扰
```

**是否影响稳定性：**

未绑定 trace 时（绝大多数场景），patch 函数直接透传原始调用，不执行任何额外逻辑，零开销。绑定 trace 后也只增加内存写入操作，不修改 LLM 的输入输出内容，**不影响调用结果**。

### Q29：DuckDBService 的智能聚合降级是怎么实现的？为什么 SUM(时间列) 不会报错？

**问题：** 如果用户在拖拽透视时，把"报警时间"（TIMESTAMP 类型）拖到值字段并选择 SUM，DuckDB 会报错（TIMESTAMP 不能 SUM）。

**解决方案：类型感知聚合：**

```python
def _make_compatible_agg(self, agg, field_name):
    t = self._get_field_type(field_name)  # 查字段类型

    if agg in ("MIN", "MAX", "COUNT"):
        return agg  # 这些对所有类型都安全

    if agg == "SUM":
        if t 是 数值类型(BIGINT/DOUBLE/FLOAT) → 返回 SUM
        if t 是 时间类型(TIMESTAMP/DATE) → 降级为 COUNT
        if t 是 字符串类型(VARCHAR) → 降级为 COUNT
        if 类型未知 → 降级为 COUNT

    if agg == "AVG":
        同理，非数值降级为 COUNT
```

**双重兜底：**

```
第一层：try 执行原始聚合
  → 如果成功 → 返回结果
  → 如果抛出 BinderException（函数签名不匹配）
第二层：将所有值字段降级为 COUNT 重新执行
```

这样用户就算把"报警时间"拖到 SUM 上，也不会看到报错，而是自动拿到计数结果。

### Q31：`run_sql_query` 工具返回前 12 行给 AI 看，如果数据有 10 万行，AI 怎么知道完整情况？

**分层返回机制：**

```python
def run_sql_query(sql, con, ...):
    df = execute_duckdb_sql(sql, con, limit=1000)  # 最多取 1000 行

    df_display = df.head(12)                         # 只给 AI 看 12 行
    df_csv = df_display.to_csv(index=False)          # CSV 格式

    return {
        "df": df,             ← 完整 DataFrame（最多 1000 行，给最终结果用）
        "csv": df_csv,        ← 前 12 行 CSV（给 AI 确认用）
        "markdown": "...",    ← 前 12 行 Markdown（给 AI 确认用）
        "sql": "SELECT ...",  ← 实际执行的 SQL
    }
```

AI 看到的只有前 12 行 CSV，用来**确认结果结构是否正确**。完整的 DataFrame（最多 1000 行）存储在 `state.query_ids[query_id].artifact["df"]` 中，最终通过 `submit_result` 提交时取出，传给前端渲染图表和透视表。

**为什么限制 1000 行？** 因为 AI 上下文窗口有限，完整的 DataFrame 可能很大。如果用户需要看所有数据，可以在聊天中说明"不限制行数"。

### Q33：Python 的 GIL（全局解释器锁）在这个系统里怎么处理的？会不会成为瓶颈？

**系统各环节的 GIL 影响分析：**

| 环节 | 是否受 GIL 影响 | 说明 |
|------|---------------|------|
| FastAPI HTTP 处理 | ✅ 不受影响 | uvicorn 使用 `asyncio` 事件循环，I/O 密集型，GIL 在 await 时释放 |
| AI Agent LLM 调用 | ✅ 不受影响 | LLM 调用是 HTTP 请求（I/O 等待），GIL 在等待时释放 |
| DuckDB 查询 | ⚠️ 部分影响 | Python ↔ C++ 边界短暂持有 GIL，但实际计算在 DuckDB 的 C++ 引擎中 |
| SSE 事件推送 | ✅ 不受影响 | 纯 I/O 操作 |
| 前端静态文件 | ✅ 不受影响 | 纯 I/O 操作 |

**实际处理方式：**

AI Agent 的 `_do_agent_pipeline()` 在后台线程池中执行：

```python
loop = asyncio.get_event_loop()
future = loop.run_in_executor(None, _do_agent_pipeline, query, session_id)
```

这样 LLM 调用（I/O 密集型）不阻塞主事件循环。SSE 推送在主协程中轮询 writer 队列，也不受影响。

**如果未来需要处理高并发：**

可以改用 `uvicorn` 的多 worker 模式（`--workers N`），每个 worker 独立进程，彻底绕开 GIL。但当前系统定位为单机工具，单进程足够。

### Q34：数据库是只读模式，那 AI 生成的 CREATE TABLE/INSERT 语句会怎么样？

**三层阻断：**

```
第一层：System Prompt 指令
  AI 被告知 "database is read-only"，但 LLM 可能不听话

第二层：DuckDB 连接配置
  duckdb.connect("vcloud_duck.db", read_only=True)
  → DuckDB 层面拒绝任何写操作

第三层：execute_duckdb_sql 的返回值处理
  def execute_duckdb_sql(sql, con, ...):
      rel = con.sql(sql)      ← 如果是 CREATE TABLE，返回 None
      if rel is None:
          return pd.DataFrame()  ← 返回空 DataFrame，不给 AI 报错
      return rel.df()
```

即使 AI 生成 `INSERT INTO` 或 `CREATE TABLE`，DuckDB 在 `read_only=True` 模式下会直接拒绝，返回异常。`run_sql_query` 工具捕获异常后以 `{"error": "..."}` 形式返回给 AI，AI 会意识到不能写并重新生成 SELECT 查询。
